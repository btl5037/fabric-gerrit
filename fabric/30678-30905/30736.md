The following is my opinion (just sharing what I feel). 

Yes, being consistently bad is not a good thing, at the same time, having a different coding style/source code structure within a package is also not a good thing. Inconsistent coding structure affects readability and extendability as well (at least in my case). 

Let's analyze the peer CLI package. Our commands are structured as "peer command sub-command". For the "command", we have a command.go (e.g., peer/chaincode/chaincode.go) and for each sub-command, we have a sub-command.go (e.g. peer/chaincode/invoke.go). 

All flags used by all sub-commands are listed in the command.go and the flagList is used to define which ones are applicable for which sub-commands. In chaincode.go, this source code structure helps to avoid duplicate flag listing. In channel.go, it does not help but it maintains the consistency with the chaincode.go. Now, in node.go, it follows a different approach (each individual sub-command defines the flag). Say that a new fabric developer wants to add a new command and a list of sub-commands. Now, that developer needs to break his head to understand why channel.go and node.go are implemented in a different way which does affect the extendability in a way. There is a possibility that the developer might introduce a new source code structure and add more inconsistency.

Hence, there is a trade-off between being consistently bad and being inconsistent. Solution: code refactoring by spending dedicated time so that we can make a consistent coding structure in a package and remove the bad behavior at once.

Other than maintainers, if we have a package level (fine granularity like peer CLI, protoutils, chaincode, etc...) un-official owners who are responsible for making the code consistent and remove the bad style by refactoring, code readability, and extendability will become easy.
