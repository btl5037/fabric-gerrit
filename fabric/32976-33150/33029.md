<strong>Project</strong>: fabric<br><strong>Branch</strong>: master<br><strong>ID</strong>: 33029<br><strong>Subject</strong>: FAB-16353 Return an error rather than ProposalResp<br><strong>Status</strong>: MERGED<br><strong>Owner</strong>: Jason Yellick - jyellick@us.ibm.com<br><strong>Assignee</strong>:<br><strong>Created</strong>: 8/20/2019, 12:52:46 AM<br><strong>LastUpdated</strong>: 8/25/2019, 3:39:38 AM<br><strong>CommitMessage</strong>:<br><pre>FAB-16353 Return an error rather than ProposalResp

Directly returning a proposal response that has an error message set and
a status of 500, embedding an error, over and over again through the
endorsement code is unweidly.  Better to work with raw errors and bubble
them up to the top.

Note, this CR causes the previously returned error (with no response) on
chaincode event marshaling errors to go back to the client.  There is no
obvious reason marshaling a chaincode event should ever fail, nor why we
would want to hide this error text from the client, so this seems like a
safe error.
