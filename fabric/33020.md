<strong>Project</strong>: fabric</br><strong>Branch</strong>: master<br><strong>ID</strong>: 33020<br><strong>Subject</strong>: FAB-16353 Stop pretending chaincode version is set<br><strong>Status</strong>: MERGED<br><strong>Owner</strong>: Jason Yellick - jyellick@us.ibm.com<br><strong>Assignee</strong>:<strong>Created</strong>: 8/20/2019, 12:52:46 AM<br><strong>LastUpdated</strong>: 8/21/2019, 9:32:19 AM<br><strong>CommitMessage</strong>:<br><pre>FAB-16353 Stop pretending chaincode version is set

The client has no idea what version chaincode they are invoking, and
they never set it.  Yet all of our endorser tests seem to believe that
they do.  Further, we build all of our metrics based on the idea that
'chaincode-id' is a property of the invocation, when in fact, it is only
chaincode name that we have access to for many of the checks.

This results in metrics that look like "mycc:" being reported as the
"chaincode" label.

If we feel that version is important to be reported for the metric, or,
if we actually want to report the chaincode id, as used internally by
the chaincode subsystem (which in the legacy case is name:version or in
the new lifecycle case is label:hash), then we can.
