> Every sys cc would have to implement this. I don't see anything fundamentally wrong in having this as an API. Is there any particular implementation/design reason that makes this removal advantageous ?
> 
> Or put it another way, is there a compelling reason that helps with the "Launch" work ?

The thread I started pulling which led me here was trying to decouple the launching/registration of a chaincode from the invocation/execution of a chaincode.  We have a lot of very goofy paths where we switch and do one thing if it's a system chaincode, and do another if it's a user chaincode.  By getting rid of the 'scc special cases', it makes these code paths more uniform and easier to refactor.

Finally, this also ties into the validation work.  Having an SCCprovider and referencing it in validation is non-deterministic and a problem, period.  So, many of these checks become superfluous, as they are artificial in nature and cannot be enforced in validation.
