<strong>Project</strong>: fabric<br><strong>Branch</strong>: master<br><strong>ID</strong>: 33403<br><strong>Subject</strong>: Index all txs without duplicate TxIDs check<br><strong>Status</strong>: MERGED<br><strong>Owner</strong>: Manish Sethi - manish.sethi@gmail.com<br><strong>Assignee</strong>:<br><strong>Created</strong>: 9/7/2019, 3:02:40 PM<br><strong>LastUpdated</strong>: 9/10/2019, 2:48:03 PM<br><strong>CommitMessage</strong>:<br><pre>Index all txs without duplicate TxIDs check

In the current code, we index only one transaction (the first one) with
a txid. This is a rare scenario that we will encounter more than one
transaction with same txid. However, this requires all over the code
to handle this situation making the code complex.

This CR, encodes the txid such that the index key for a txid becomes
unique and can simply be added to the index without worrying about
actual duplicates. The encoding leverages the block number and
transaction number to achieve this.

During lookup, first such entry for a txid (in practice, there should be
only one such entry) is retuned which maintains the current semantics
(i.e., queries return only first transaction with a given txid).
However, the new index would be able to support another query that
returns all the transactions with a given query in future.

Because of above mentioned new encoding, this requires a rebuild of
block storage index on the peer
