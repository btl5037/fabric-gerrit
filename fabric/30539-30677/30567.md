<strong>Project</strong>: fabric<br><strong>Branch</strong>: master<br><strong>ID</strong>: 30567<br><strong>Subject</strong>: [FAB-14913] cc2cc with same package<br><strong>Status</strong>: ABANDONED<br><strong>Owner</strong>: Alessandro Sorniotti - ale.linux@sopit.net<br><strong>Assignee</strong>:<br><strong>Created</strong>: 3/31/2019, 10:40:11 AM<br><strong>LastUpdated</strong>: 5/9/2019, 8:26:58 AM<br><strong>CommitMessage</strong>:<br><pre>[FAB-14913] cc2cc with same package

Problem: the new lifecycle decouples the identifiers of a chaincode instance
from the identifiers of the chaincode package/container. As such, it is
possible to instantiate the same package (backed by the same container)
multiple times on the same channel. Normally this does not cause any
problems: the peer discovers the chaincode container from the registry,
creates a transaction context (indexed by channel/txid) and uses it to
perform the invocation. However, it is entirely legal for a chaincode to have
to invoke another instance of itself with a different name on the same
channel (case in point, this is the scenario of the state based endorsement
integration test in integration/sbe). When this happens however, the
invocation fails because the transaction context (indexed by channel/txid)
already exists in the internal map of transaction contexts, since the context
for the chaincode-to-chaincode invocation and the context for the original
invocation have the same key, since channel and txid are identical.

Fix: the map of transaction contexts maps channel/txid to a stack of
transaction contexts (no longer a single one). Normally the stack has depth
1. The stack may have depth >1 in case the same chaincode happens to invoke
another instance of itself on the same channel. This is safe because
chaincode to chaincode invocations behave like function calls, where
contexts are nested: the context of the caller is guaranteed to be created
strictly before the callee is created and to be destroyed strictly after
the callee is destroyed.

Tests: aside from unit tests, a new integration test has been created.
