<strong>Project</strong>: fabric<br><strong>Branch</strong>: master<br><strong>ID</strong>: 25299<br><strong>Subject</strong>: FAB-11506 configtxlator do not deeply encode nil<br><strong>Status</strong>: MERGED<br><strong>Owner</strong>: Jason Yellick - jyellick@us.ibm.com<br><strong>Assignee</strong>:<br><strong>Created</strong>: 8/8/2018, 2:19:24 PM<br><strong>LastUpdated</strong>: 8/10/2018, 4:46:25 PM<br><strong>CommitMessage</strong>:<br><pre>FAB-11506 configtxlator do not deeply encode nil

The protolator code used by configtxlator checks to see if the proto
message contains opaque fields (ie, byte slices which are really proto
messages).  However, sometimes, a field is deliberately omitted, and
there is not enough information to determine the encoding of the field,
unless it is populated.  This can result in errors encoding sparse
messages as protobuf code is told to emit defaults.

This CR adds a check before attempting to deeply marshal a field (be it
plain, map, or slice), that if the field is nil, it should not be
deeplly marshaled.

This addresses a bug seen decoding channel configs which embed the
sparse config update structure.
