I think there is a much better way of implementing cache for an MSP:
Essentially, an MSP is an object that only can be mutated via the Setup() function (which... anyway is called in the code right after the MSP creation, but lets put that aside).

Now, what if we build a cache in the following way:
- Instead of adding cache objects into the bccsp-msp, we just build a cache around the MSP by having an object that embeds an MSP, and just overrides some of its methods.
Then, the object would just do a cache lookup for the arguments of methods that are invoked a lot, like DesrializeIdentity, Validate, and satisfiesPrincipal and returns cached computation results.
We are guaranteed that as long as Setup() isn't called - the computation results will be the same each time. 

Now, this not only gives up the ability to support ANY MSP (instead of just bccsp-msp) but also gives us modularity, and an ability to easily test the cache by mocking the MSP interface.</pre><strong>Reviewer</strong>: Angelo De Caro - adc@zurich.ibm.com<br><strong>Reviewed</strong>: 8/23/2017, 8:07:36 PM<br><strong>Message</strong>: <pre>Patch Set 3: Patch Set 2 was rebased</pre><strong>Reviewer</strong>: Hyperledger Jobbuilder - jobbuilder@jenkins.hyperledger.org<br><strong>Reviewed</strong>: 8/23/2017, 8:07:43 PM<br><strong>Message</strong>: <pre>Patch Set 3:
